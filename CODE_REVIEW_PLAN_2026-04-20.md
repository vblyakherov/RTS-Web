## Findings

### 🔴 Критично / безопасность

#### 1. JWT в `localStorage` + CSP разрешает `'unsafe-inline'`

`frontend/js/api.js` хранит `access_token` в `localStorage`. При этом CSP в `nginx/nginx.conf` разрешает:

```text
script-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net
```

Следствие:

- любой XSS может украсть токен;
- инлайн-скрипты во всех HTML-файлах не дают убрать `unsafe-inline` без отдельного фронтенд-прохода.

Рекомендация:

- перевести браузерную сессию на `httpOnly Secure SameSite` cookie;
- добавить CSRF-защиту для mutating requests;
- поэтапно вынести инлайн-скрипты страниц во внешние файлы;
- после этого убирать `unsafe-inline` из CSP.

#### 2. Экранирование в шаблонных строках фронта ненадёжно

Паттерн из `regions.html`:

```js
onclick="openDeleteModal(${r.id}, '${escHtml(r.name).replace(/'/g,"\\'")}')"
```

Проблема:

- `escHtml` экранирует только `&`, `<`, `>`;
- значение вставляется в JS-литерал внутри HTML-атрибута;
- это анти-паттерн HTML + JS-интерполяции и потенциальная XSS-точка.

Рекомендация:

- уходить от `onclick` в строках;
- использовать `addEventListener`, `data-*`, `textContent`;
- не строить исполняемый JS из пользовательских значений.

#### 3. Токен экспорта Excel живёт 7 дней

Текущее значение:

```python
EXCEL_EXPORT_TOKEN_EXPIRE_MINUTES: int = 10080
```

Проблема:

- токен встраивается в XLSM/VBA-контур;
- это полноценный JWT пользователя;
- если файл попадёт к третьему лицу, оно сможет выполнять запросы от имени пользователя, включая `POST /api/v1/sync`.

Минимальная рекомендация:

- вводить отдельный `scope` или `token_type`, например `excel_sync`;
- проверять его в `/sync` и `/sync/columns`;
- добавлять в payload хотя бы `project_id`, а при необходимости и contractor-scoping.

#### 4. Rate limit есть только на `/login`

Сейчас лимит стоит только на логине:

```python
@limiter.limit("5/minute")
```

Проблемы:

- `slowapi` с `get_remote_address` за nginx видит не реальный IP клиента, а upstream/proxy IP;
- все клиенты могут делить один лимит;
- отсутствует лимит на тяжёлые пути вроде `/sync` и `/excel/import`.

Рекомендация:

- заменить key function на чтение `X-Forwarded-For` или `X-Real-IP`;
- добавить лимиты на `/sync` и `/excel/import`.

#### 5. `APP_ENV` и `SECRET_KEY`

Наблюдения:

- `APP_ENV=production` по умолчанию выглядит как безопасный дефолт;
- `SECRET_KEY` без fallback валит приложение при старте, если конфигурация не задана, и это приемлемо;
- при этом нет явной валидации минимальной длины или качества секрета.

Рекомендация:

- добавить проверку минимальной длины или базовых требований к `SECRET_KEY`.

#### 6. Тайминг логина раскрывает существование пользователя

Текущий паттерн:

```python
if not user or not verify_password(...):
    raise HTTPException(401, "Incorrect username or password")
```

Проблема:

- если пользователь не найден, хэш не проверяется;
- это даёт небольшую тайминг-разницу.

Рекомендация:

- при отсутствии пользователя всё равно прогонять dummy password verify по заранее известному хэшу.

### 🟠 Функциональные баги / подозрительные места

#### 7. `sync_reference_directories(db)` на каждом `GET /sites` и `GET /sites/{id}`

Проблема:

- синхронизация справочников гоняется на read-path;
- происходят `SELECT DISTINCT`, возможные `INSERT`, `UPDATE`, `flush`;
- запись в БД идёт внутри GET-запросов;
- это нарушает семантику чтения и масштабируется плохо.

Рекомендация:

- делать справочную синхронизацию только на write-path;
- либо вызывать её on-demand через отдельный admin action или cron.

#### 8. `sync_contractors_from_sites` стирает ручные изменения

Проблема:

- `is_active` подрядчиков перезаписывается автоматически на основе сайтов;
- PATCH в UI фактически может быть тут же откатан следующими read-path вызовами;
- это ломает ожидаемую админскую семантику.

Рекомендация:

- либо признать `is_active` полностью автополем и убрать ручное управление;
- либо признать его ручным полем и убрать авто-перезапись.

#### 9. `rollback-entry` не всегда совпадает с пользовательским ожиданием "отката"

Текущая логика берёт `entry.old_value` и пишет его в поле.

Проблема:

- для ранних исторических записей это скорее "установить старое значение из этой записи", чем "откатить текущее состояние";
- семантика отличается от rollback по timestamp.

Рекомендация:

- либо поменять UI copy и назвать действие честнее;
- либо изменить серверную семантику.

#### 10. `_coerce_value` потенциально хрупок по порядку веток

Проблема:

- коэрсинг зависит от `target_type` и `field_name`;
- при неаккуратном изменении реестра поведение может сломаться молча.

Рекомендация:

- постепенно переходить к именованным коэрсерам по `db_name` с fallback по типу.

#### 11. `_clean_row` молча проглатывает пустую после очистки строку

Проблема:

- если после очистки/коэрса строка опустела, она превращается в `None`;
- вызывающий код делает `continue` без ошибки и без явного сигнала клиенту.

Рекомендация:

- фиксировать ошибку вроде "строка не содержит изменяемых полей" вместо немого пропуска.

#### 12. `bulk_upsert_sites` больше не создаёт сайты, но всё ещё возвращает `created`

Проблема:

- `created` фактически всегда `0`;
- код и ответ API вводят в заблуждение.

Рекомендация:

- убрать мёртвую метрику или честно переименовать результат.

#### 13. Двойное хранение региона: `region` и `region_id`

Проблема:

- есть денормализация;
- часть кода работает по строковому полю, часть по FK;
- нужен дополнительный helper `_sync_region_text`;
- есть риск рассинхрона и лишние запросы.

Рекомендация:

- долгосрочно оставить один источник истины, предпочтительно FK `region_id`.

#### 14. И `get_db`, и эндпоинты коммитят транзакции

Проблема:

- часть эндпоинтов делает `await db.commit()` явно;
- `get_db` тоже коммитит в dependency lifecycle;
- это смешивает контракты транзакционности.

Рекомендация:

- выбрать одну модель;
- предпочтительно: коммит только в mutating endpoints и никогда в read dependency.

#### 15. `get_db` коммитит даже на GET

Проблема:

- любой успешно завершённый GET завершался `commit`;
- из-за write-on-read это приводило к "невидимым" записям.

Рекомендация:

- не коммитить в dependency;
- мутирующие пути должны коммитить явно.

#### 16. Нет пагинации в `get_regions` и `get_contractors`

Проблема:

- со временем эндпоинты будут отдавать весь справочник целиком.

Рекомендация:

- добавить пагинацию или хотя бы limit/offset.

#### 17. `Site.project_id` допускает `NULL`

Проблема:

- на уровне модели `project_id` nullable и `ondelete='SET NULL'`;
- большая часть доменной логики уже предполагает, что сайт всегда принадлежит проекту.

Рекомендация:

- либо запретить `NULL` и сменить контракт удаления проекта;
- либо явно обрабатывать orphaned sites.

#### 18. `require_any = require_roles(*UserRole)` семантически неочевиден

Проблема:

- имя звучит слишком абстрактно;
- будущая новая роль автоматически попадёт в доступ.

Рекомендация:

- использовать более явный контракт для "любой авторизованный".

### 🟡 Архитектура / качество

#### 19. Модель `Site` и реестр колонок живут как два источника правды

Проблема:

- `Site` содержит огромной объём вручную объявленных колонок;
- параллельно есть `SITE_COLUMNS`;
- изменение схемы требует правок в нескольких местах.

Рекомендация:

- либо приблизить модель к генерации из реестра;
- либо рассмотреть частичный перенос template-specific части в JSONB.

#### 20. CRUD-роутеры во многом копипастны

Проблема:

- `regions.py`, `contractors.py`, `projects.py` сильно похожи;
- добавление новых сущностей раздувает boilerplate.

Рекомендация:

- думать о generic CRUD router/factory.

#### 21. В sync-слое есть отложенные импорты и "приватные" функции как межмодульное API

Проблема:

- есть признаки циклических зависимостей;
- `_coerce_value` используется извне, хотя обозначен как внутренний.

Рекомендация:

- вынести общий коэрсинг в отдельный модуль без циклов.

#### 22. Frontend дублирует словари и labels с backend

Проблема:

- `STATUS_LABELS`, `ROLE_LABELS` и т.п. живут и на бэке, и на фронте;
- любое изменение требует синхронной правки в двух местах.

Рекомендация:

- со временем отдавать метаданные с backend.

#### 23. Во frontend нет типизации

Проблема:

- масштаб логики уже высок;
- чистый JS упрощает внедрение регрессий.

Рекомендация:

- как минимум рассмотреть `checkJs: true` и JSDoc;
- дальше возможен мягкий путь в сторону TypeScript.

#### 24. `login.html` редиректит только по наличию токена

Проблема:

- невалидный токен даёт лишний редирект на `/index.html` и обратный возврат.

Рекомендация:

- проверять не только наличие токена, но и валидность сессии.

#### 25. Security headers: нет HSTS

Проблема:

- сейчас сайт ещё на HTTP;
- после перехода на HTTPS легко забыть включить HSTS.

Рекомендация:

- включить `Strict-Transport-Security` при полноценном HTTPS rollout;
- deprecated `X-XSS-Protection` можно будет убрать.

#### 26. `excel.py` использует несколько Excel-стеков

Наблюдение:

- экспорт через `xlsxwriter`;
- импорт и часть операций через `openpyxl`;
- в проектной документации фигурируют и другие инструменты.

Рекомендация:

- явно документировать ответственность каждого инструмента.

#### 27. Логи проектов пишут `extra=data.model_dump(...)` без sanitizer'а

Проблема:

- если в `ProjectUpdate` появится чувствительное поле, оно уйдёт в логи.

Рекомендация:

- делать allowlist или единый sanitizer для audit payloads.

#### 28. `Request = None` в `excel_export`

Проблема:

- формально FastAPI это переварит;
- но для сигнатуры и типизации это плохой паттерн.

Рекомендация:

- сделать `Request` обязательным параметром без default.

#### 29. Клиентская обработка ошибок пока слишком "строковая"

Проблема:

- `showToast(err.message, 'danger')` нормален для текущих строковых ошибок;
- но плохо масштабируется под structured errors.

Рекомендация:

- со временем ввести клиентский `AppError` с `code/message/details`.

## Что уже было отмечено как сильные стороны

- Чистое разделение слоёв: API → CRUD → models → services.
- Фабрика `require_roles` читается хорошо.
- История изменений с `sync_batch_id` выглядит продуманно.
- Rate limit на login уже есть.
- Security headers вынесены в nginx, а не смешаны с приложением.
- `bcrypt==4.0.1` зафиксирован осознанно.
- `SITE_COLUMNS` уже задаёт правильное направление к единому реестру полей.

## Короткий TOP-5 к починке

1. Убрать скрытые коммиты из dependency-контракта и не смешивать их с ручными commit.
2. Убрать write-on-read через справочную синхронизацию на GET.
3. Определиться с ручной или автоматической природой `is_active` у справочников.
4. Ограничить Excel-JWT по scope и проекту, а rate limit привязать к forwarded IP.
5. Уйти от `localStorage` + `unsafe-inline` к cookie-based auth и жёсткой CSP.

## План исправления

Ниже сохранён исходный план исправлений, который был предложен после review.

### Этап 1. Убрать скрытые записи из read-path

Цель:

- вынести `session.commit()` из `get_db`;
- оставить только явные коммиты в mutating endpoints;
- убрать `sync_reference_directories()` из GET-роутов;
- прекратить автоперезапись `is_active`.

Главный риск:

- забыть один из write-endpoints и потерять запись.

Тест-стратегия:

- узкие backend-тесты на `auth`, `sites`, `regions`, `contractors`, `sync`.

Статус:

- уже сделано.

### Этап 2. Починить Excel auth-контур

Цель:

- ввести отдельный `token_type` или `scope=excel_sync`;
- добавить в payload хотя бы `project_id`, а лучше ещё и ограничение по contractor;
- проверять это в `/sync` и `/sync/columns`;
- параллельно заменить limiter key на forwarded IP и добавить лимиты на `/sync` и `/excel/import`.

Статус на 2026-04-21:

- этап `2A` выполнен;
- `GET /api/v1/excel/export` теперь встраивает отдельный `token_type=excel_sync`;
- токен несёт `project_id` и, при наличии, `contractor_id`;
- обычные browser-endpoints принимают только `access` token;
- `/sync` и `/sync/columns` принимают `excel_sync`;
- `/sync` отклоняет project mismatch и строки с `site_id` из другого проекта;
- подтверждённый server-side backend regression: `91/91` на VPS.

### Этап 3. Закрыть реальные XSS-синки

Цель:

- перестать собирать `onclick`, `innerHTML` и `insertAdjacentHTML` из сырых строк;
- начать с `utils.js`, `regions.html`, `contractors.html`, `site.html`, `sites.html`, `reports.html`.

Предпочтительный подход:

- `addEventListener`;
- `data-*`;
- `textContent`;
- не плодить новый слой "экранирующих" helper’ов как основную защиту.

### Этап 4. Перевести сессию с `localStorage` на cookie

Цель:

- `httpOnly` + `Secure` + `SameSite`;
- CSRF для mutating requests;
- адаптация `api.js` под cookie-based auth.

Комментарий:

- это лучше делать после этапа 3, потому что XSS всё равно сможет действовать от имени пользователя, даже если не сможет прочитать токен.

### Этап 5. Дожать CSP и заголовки

Цель:

- вынести inline-скрипты страниц во внешние файлы;
- убрать `'unsafe-inline'`;
- после появления HTTPS включить HSTS.

### Этап 6. Второй эшелон cleanup

Сюда относятся:

- `rollback-entry` semantics и UI copy;
- `_clean_row` с немым `continue`;
- мёртвый `created`;
- пагинация справочников;
- rename `require_any`;
- затем более тяжёлые архитектурные темы вроде денормализованного `region` и монолитной модели `Site`.

## Рекомендуемый порядок следующих этапов после завершённого этапа 1

После того как этап 1 уже закрыт, продолжать работу рекомендуется не "одной security-мегаветкой", а отдельными управляемыми кусками.

### 2A. Excel scoped token

Фокус:

- сузить долгоживущий XLSM-токен до отдельного типа `excel_sync`;
- добавить `project_id`;
- при необходимости добавить contractor-scoping;
- валидировать ограничения в `/sync` и `/sync/columns`.

Почему это следующий лучший шаг:

- высокий security-value;
- небольшой blast radius;
- почти не требует фронтенд-переезда.

Риск:

- сломать уже существующий XLSM-поток или contractor-scoping.

Тесты, которые стоит обновить первыми:

- `Tests/backend/test_sync.py`
- `Tests/backend/test_excel.py`

Что проверить:

- позитивный sync с excel-токеном;
- отказ для token scope mismatch;
- отказ при неверном `project_id`;
- сохранение корректного доступа для штатного Excel-сценария.

### 2B. Proxy-aware rate limiting

Фокус:

- перейти с `get_remote_address` на forwarded-IP key function;
- добавить лимиты на `/sync` и `/excel/import`.

Почему лучше отдельно от 2A:

- проще откатить и настраивать политику лимитов, не трогая auth-контур.

Риск:

- задушить реальных пользователей или shared NAT;
- неверно доверять proxy headers.

Тесты, которые стоит обновить первыми:

- узкие backend-тесты на key-function;
- проверка `429` для `/login`, `/sync`, `/excel/import`.

### 3A. Убрать XSS-синки на простых страницах

Фокус:

- начать с `regions.html` и аналогичных страниц со схожим CRUD-паттерном;
- убрать строковые `onclick`;
- перейти на DOM handlers.

Почему так:

- это одновременно режет XSS-риск и подготавливает переход к CSP без `unsafe-inline`.

Риск:

- сломать модалки и controlled reload UX.

Тесты, которые стоит обновить первыми:

- `Tests/frontend/test_utils.test.js`
- при необходимости новые frontend unit-tests под вынесенные page-scripts.

### 3B. Дожать XSS-синки на сложных страницах

Фокус:

- `sites.html`
- `site.html`
- `reports.html`
- участки `utils.js` с `innerHTML` и `insertAdjacentHTML`

Риск:

- большой фронтовый слой;
- лучше идти сюда только после обкатки паттерна на простых страницах.

Тестовая поверхность:

- targeted Jest;
- ближайший e2e/smoke по списку объектов, карточке и отчётам.

### 4. Browser auth: dual-stack переход на cookie

Фокус:

- временно поддерживать и cookie, и текущий bearer-контур;
- затем переводить браузерный фронт на httpOnly cookie;
- добавить CSRF для mutating routes;
- убрать зависимость браузера от `localStorage`.

Почему не раньше:

- до закрытия основных XSS-точек browser-auth migration не даёт полный эффект.

Риск:

- затронуть весь auth flow, logout, profile, project selection и все защищённые страницы.

Тестовая поверхность:
